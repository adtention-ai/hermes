# Changelog

## [Unreleased]

**Client-side anti-farming hardening.**

### What changed

- Dedupes render acknowledgments by `impression_id`, not by mutable platform message ID.
- Consumes the cached sponsor after the first successful render acknowledgment attempt.
- Adds a short sponsor cache TTL so stale served impressions are not reused indefinitely.
- Caps sponsored renders to one impression per gateway turn.
- Requires an explicit gateway status path or status metadata before decorating `send`/`edit_message`, preventing prefix-spoofed final messages from being sponsored.
- Stops treating tool/debug progress messages as billable wait-state sponsor surfaces.
- Documents why Telegram/Discord delivery success is not proof the user saw the message, and how clicks/subsequent activity can be used only as weak fraud/risk signals.

## [0.1.2] - 2026-06-26

**Corrupt local cache recovery.**

This patch release makes the Hermes plugin resilient when its local SQLite state file is corrupt or replaced with non-SQLite bytes. Instead of crashing during startup, the plugin quarantines the bad cache file and recreates a fresh database.

### What changed

- Detects SQLite startup failures caused by a malformed or non-database local `adtention.sqlite3` file.
- Moves the corrupt file aside as `adtention.sqlite3.corrupt-*.bak` for inspection instead of deleting it.
- Recreates a clean state database so Hermes can keep loading the ADtention plugin.
- Adds regression coverage for the corrupt-database recovery path.

### Install / update

```bash
hermes plugins install adtention-ai/hermes --enable
hermes gateway restart
```

ADtention for Hermes still targets recognized Telegram/Discord wait states only.

### Privacy

No privacy model changes. The quarantined cache is local-only and is never sent to ADtention. The plugin still never sends prompts, replies, code, chat IDs, tool arguments, terminal output, or tool output.

### Release verification

- `python -m compileall adtention_hermes` passes.
- `ruff check .` passes.
- `git diff --check` passes.

## [0.1.1] - 2026-06-24

**Hermes client attribution + launch polish.**

This is the launch-ready Hermes plugin release: Hermes traffic is now explicitly tagged for the ADtention backend, and the public docs/release process now match the quality bar of the OpenCode plugin.

### What changed

- Tags `POST /v1/register` and `POST /v1/serve` with `client: "hermes"` so the backend can attribute impressions and publisher registrations to the Hermes surface.
- Keeps the client tag static and non-identifying. It is product attribution only, not user data.
- Adds privacy tests guarding that the new field is present and still constrained by outbound payload allowlists.
- Rewrites the README around the user promise, privacy model, earning mechanics, install flow, FAQ, and clean uninstall path.
- Adds curated release notes through this changelog and updates the release workflow to publish the matching section instead of low-signal autogenerated notes.
- Updates package/plugin metadata language for a clearer GitHub/package description.

### Privacy

The `client` field does not change the privacy model. ADtention for Hermes still never sends prompts, replies, chat history, code, files, filenames, paths, repo names, chat IDs, user IDs, tool arguments, terminal output, or tool output.

### Install / update

```bash
hermes plugins install adtention-ai/hermes --enable
hermes gateway restart
```

Then run from Telegram or Discord:

```text
/adtention status
/adtention privacy
```

### Release verification

- Targeted client-tag tests pass.
- Full test suite passes.
- `python -m compileall adtention_hermes` passes.
- `ruff check .` passes.
- `python -m build` produces wheel and sdist artifacts.

### Known limitations

- Requires a Hermes gateway restart after install/update.
- Currently decorates recognized Telegram/Discord wait-state messages only.
- Cashout is not in this plugin; payout/account flows will live in ADtention’s backend/product surface.

## [0.1.0] - 2026-06-24

**First public Hermes wait-state release.**

ADtention for Hermes adds one quiet sponsor line to Telegram/Discord wait-state messages while your agent works.

### Highlights

- Decorates recognized Hermes wait-state/status bubbles only.
- Never modifies final assistant answers and never sends standalone ads.
- Classifies broad task intent locally; prompts, replies, chat history, files, paths, repo names, chat IDs, user IDs, tool arguments, terminal output, and tool output never leave the machine.
- Fetches sponsors in the background, at most once every 15 seconds, so rendering does not wait on the ADtention API.
- Acknowledges impressions only after a decorated wait-state message is successfully sent or edited.
- Adds `/adtention status`, `/adtention on`, `/adtention off`, `/adtention privacy`, and `/adtention sponsor`.
- Stores local identity, cached sponsor, balance, and settings under the active Hermes profile.
- Runs stdlib-only at runtime.

### Install

```bash
hermes plugins install adtention-ai/hermes --enable
hermes gateway restart
```

### Privacy

The plugin sends only broad category metadata, pseudonymous install/render identifiers, host/surface/platform labels, and version/client metadata. It never sends prompts, replies, code, files, filenames, paths, repo names, chat IDs, user IDs, tool arguments, terminal output, or tool output.

### Known limitations

- Requires a Hermes gateway restart after install/update.
- Currently decorates recognized Telegram/Discord wait-state messages only.
- Compatibility mode wraps gateway adapter send/edit methods because Hermes did not require a core PR for this release.
